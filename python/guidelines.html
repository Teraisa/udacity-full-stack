Pep8:

Python's style guide Pep8370 is used to make sure that we write consistent, readable, and maintainable code.

There are a number of things to check, and I've detailed a few of the most common here:


  • The maximum line length78 is 80 characters. It's common to be programming and write a line as such: 

my_object = objectConstructor("This is an argument that is a really really long string that defines something, but goes past 80 characters......", argument2, argument3)


As you can see, you have to scroll horizontally to read the code, and this dimenishes readability. Lets refactor:

new_object = objectConstructor("This is an argument that is a really really"
                               " long string that defines something, but stays"
                               " under 80 characters......",
                               argument2, argument3)

Hopefully, this is evident in being more easily read than the first example.

Note that urls are an exception here. Don't split a url over two lines, instead add the comment # NOQA662 to the end to suppress linter warnings.


In other cases the method signature --the myMethod(arg1, arg2, etc) bit of code-- or a dict or array declaration passes 80 characters. You can resolve this by moving arguments to a new line as such:

# With a method signature:
new_object.myMethod(long_argument1, long_argument2, long_argument3,
                    long_argument4, long_argument5, etc...)

# And with an array:
my_array = [long_array_element1, long_array_element2, long_array_element3,
            long_array_element4, long_array_element5, etc...]


  • Whitespace near comments:

It's good practice to document your code with comments, and block and inline comments are great places to start. However, the following will throw two errors.

#My really awesome comment that needs a space
if foo < bar:# This inline comment is too close.
    print "Hello world!"

Lets refactor as:

# My awesome comment that now has a space between the # symbol and text
if foo < bar:  # This inline comment has 2 spaces now!
    print "Hi again!"



There are many other aspects of writing pep8 compliant code, but these are the most common. Be sure to watch this great PyCon talk101 on Pep8 by Raymond Hettinger and see the PyGuide64 for more information.

As this can be tedious work, check out pep8online143 to quickly check a snippet and the tool autopep8132 to catch and fix any errors for your entire project. Also, if you use sublime text as your editor your can integrate this into your workflow via package control.


README:

Readmes are a central piece of documentation for your project. They are often the first place that a user will look when coming to your codebase.

  • Did you include a readme?
  • Does this readme give the user instructions on how to get started running your python code to generate their own HTML page?

Here are a few resources that will get you started writing great readmes:
- Udacity's Writing READMEs class692
- Stack Overflow: How to write a good README586
- Awesome-readme repo317. Examples of great readmes on Github.
- Blog Post: How to Write a Readme Worth Reading217



Note: Markdown is a very helpful format that makes displaying your readme easier, and it is typically used for readmes.

To format your readme as a markdown file, you'll need to save it as readme.md (with a .md file extension). See Github's guide120 and this cheat sheet154 for more on how to use markdown. Also check out dillinger.io76 to view and edit markdown in your browser.

Comments:

We've already mentioned comments, but let's go into more detail.

Comments are an important element of any readable and maintainable codebase. Often we will use these to tell other developers what the following section of code does or how it does it.

  • Block and inline comments: (PyGuide here43)

class IOTask:
    def terminate(self):
        self._running = False

    def run(self, sock):
        # sock is a socket
        sock.settimeout(5)        # Set timeout period
        while self._running:
            # Perform a blocking I/O operation w/ timeout
            try:
                data = sock.recv(8192)
                break
            except socket.timeout:
                continue
            # Continued processing
            ...
        # Terminated
        return

This code is taken from page 487 of the Python Cookbook22, and while what the code is doing isn't relevant here, take a look at the use of comments. These make it relatively easy to have a cursory grasp of what the code is doing just at a glance.


  • DocStrings: (Pep8 documentation43)
    Docstrings are a special kind comment. Their advantage is that a class's or method's docstring can be accessed via the __doc__ attribute. For instance:

class myClass(object):
    """This is a example showing docstrings
    
    Attributes:
        attr1 (str): This should say something about attr1.
        attr2 (int): This should tell the purpose of attr2 
    
    Etc....
    """

    self.attr1 = "My String!"
    self.attr2 = 42

print myClass.__doc__

would output:

This is a example showing docstrings
    
    Attributes:
        attr1 (str): This should say something about attr1.
        attr2 (int): This should tell the purpose of attr2 
    
    Etc....

This can also be done on methods, and the use of docstrings is super helpful in readability as well as debugging.

Check out this resource68 if you would like to see addition examples of docstrings.


Testing:

Testing is one tool that software developers use to ensure their code does what is supposed to. There are many approaches and methodologies to this. In our case, we simply should ensure that running our code has the desired effect of generating an html page. Definitely run your code to ensure that it works.

As the applications you develop evolve, your testing tools should too. See Udacity's Testing Software139 course and Engineering higher quality through
agile testing practices77 by Atlassian for more.


By the way: Once you receive feedback from your review, it'll come in two sections

  • Project Review: this is where you find general feedback about your project and things that might not be specific to a particular line(s) of code.

  • Code Review: Feedback on specific areas of code. ("Required", "Suggestion", or "Awesome") This is where some of the most useful points of improvement are often located.
